1. EXECUTIVE SUMMARY
═══════════════════════════════════════════════════

MedBrain is an open-source Hospital Management System built for Indian hospitals and medical colleges. "The Brain Behind Your Hospital."

This RFC defines the complete system architecture covering dual-database design, authorization model, encryption strategy, privacy controls, communication system, conflict detection, and medical college integration.

Core differentiators:
- Dual-database architecture (YottaDB + PostgreSQL) modeled after Epic/Cerner
- Attribute-Based Access Control (ABAC) with context-aware data access lifetimes
- 7-layer encryption with per-patient key hierarchy
- Address-based staff-patient conflict detection
- Masked communication system (Uber model for hospital calls)
- Privacy-first notification system (numbers only on public channels)
- Medical college academic integration (NMC compliant)
- Default NABH/ABDM/DPDPA compliance — not add-on modules
- Open source (AGPL) with commercial support tiers
- Rust + Axum backend — 10-20x faster than Java/.NET competitors

Target market: 766 medical colleges, 70,000+ hospitals in India. International expansion to GCC, Africa, South Asia.


2. DUAL-DATABASE ARCHITECTURE
═══════════════════════════════════════════════════

2.1 Why Two Databases
─────────────────────

Epic Systems — the world's largest EMR — runs on InterSystems Caché (MUMPS/M family, same as YottaDB). Their architecture uses Chronicles (Caché for real-time clinical) + Clarity (SQL Server for analytics/reporting with nightly ETL). This is not accidental. Hierarchical databases are naturally suited for patient data because clinical data IS hierarchical: Patient → Visits → within each visit: Vitals, Orders, Notes, Prescriptions.

Getting a full patient record in SQL requires 8+ table JOINs across patients, encounters, vitals, lab_orders, lab_results, prescriptions, diagnoses, and allergies. In YottaDB, the same operation is a single tree traversal of ^EPT("ACMS-2026-001234") — walking from root to leaves. The database structure matches clinical reality.

YottaDB runs IN-PROCESS — no separate database server, no network hop, no connection pool, no TCP overhead. It loads as a library inside the Rust process and provides direct memory access. This is comparable to SQLite's deployment model but with ACID transactions, journaling, encryption at rest, and replication built in.

2.2 Database Responsibility Matrix
───────────────────────────────────

Layer               | Database          | What's Stored
--------------------|-------------------|------------------------------------------
Patient clinical    | YottaDB           | Patient records, encounters, vitals,
                    |                   | orders, results, notes, prescriptions
Master files/config | YottaDB           | Form definitions, lab test catalog,
                    |                   | drug formulary, charge rates, department
                    |                   | config, templates, module config
Billing & finance   | PostgreSQL        | Invoices, payments, TPA claims, GL
                    |                   | entries, financial reports
ABAC & security     | PostgreSQL+Redis  | Policies, roles, sessions, access cache
Audit trail         | PostgreSQL        | Immutable append-only access log
Analytics/MIS       | PostgreSQL        | ETL'd from YottaDB nightly + real-time
                    |                   | dashboards via Octo SQL-on-YottaDB
Search              | Meilisearch       | Patient search, drug search, ICD-10
                    |                   | (fuzzy, typo-tolerant)
DICOM images        | Orthanc           | Medical images (X-ray, CT, MRI, USG)
Files               | MinIO             | Scanned documents, photos, attachments

2.3 Master Files Pattern
────────────────────────

Inspired by Epic's Chronicles filing cabinet model. Master files store all configurable data as hierarchical trees:

  ^MF("EPT","ITEMS")          → Patient registration form fields
  ^MF("LAB","TESTS")          → Lab test catalog (CBC, LFT, RFT...)
  ^MF("LAB","PANELS")         → Lab panels (Fever workup = CBC+MP+Widal)
  ^MF("DRUG","FORMULARY")     → Drug catalog (names, doses, interactions)
  ^MF("DRUG","INTERACTIONS")  → Drug-drug interaction rules
  ^MF("CHARGE","RATES")       → Procedure/service charge rates
  ^MF("DEPT","CONFIG")        → Department configuration
  ^MF("FORM","TEMPLATES")     → Clinical note templates
  ^MF("ICD10","CODES")        → Diagnosis code lookup
  ^MF("MODULE","STATUS")      → Module enable/disable flags
  ^MF("ENCRYPTION","RULES")   → Field-level encryption configuration

Key property: Admin can add a field to the patient registration form, add a new lab test to the catalog, or change a charge rate WITHOUT any code changes, deployment, or restart. The React frontend fetches form definitions from master files at runtime and renders dynamically. This means every hospital can customize MedBrain to their workflow without developer involvement.

2.4 Module Enablement
─────────────────────

All 67 modules are deployed in every installation. Modules are activated by configuration, not code deployment.

Example — enabling the Dental module:
1. Admin sets ^MODULE("DENTAL","STATUS")="ACTIVE"
2. System creates department ^DEPT("DENTAL")
3. Admin creates roles: Dentist, Dental_Nurse, Dental_Receptionist
4. ABAC policy templates applied from ^POLICY_TEMPLATE("DENTAL")
5. Dental master files configured: ^MF("DENTAL_PROCEDURE"), ^MF("DENTAL_CHART")
6. Dental menu items appear in navigation
7. Zero code changes. Zero deployment. Configuration only.

2.5 Data Flow Architecture
──────────────────────────

Three search paths serve different needs:

Path 1 — Meilisearch (human search, primary):
  User types in search box → Meilisearch handles fuzzy, typo-tolerant search
  Synced from YottaDB via NATS events on every patient create/update
  Latency: 5-15ms including typo correction
  Example: "Rav Kumr" finds "Ravi Kumar" (MocDoc cannot do this)

Path 2 — YottaDB Direct (exact lookup):
  System needs specific record → direct global access
  UHID/MRN/phone number → O(1) microseconds, in-process
  No network, no query parsing, no query plan — direct memory access
  Used for: record opening, encounter loading, vitals retrieval

Path 3 — PostgreSQL/Octo (analytics):
  Complex queries, aggregations, reports, MIS dashboards
  Octo provides PostgreSQL wire protocol over YottaDB data
  Compiles SQL to M routines running as native machine code
  Used for: revenue reports, department stats, NABH indicators

Form rendering flow:
  GET /api/v1/masterfiles/EPT/form-definition
  → YottaDB returns ^MF("EPT","ITEMS") as JSON
  → React renders form dynamically based on JSON schema
  → Hospital admin adds field → form updates without deployment

2.6 Event-Driven Architecture
─────────────────────────────

Traditional HMS (MocDoc etc.): Doctor orders lab → saves to database → done. Lab tech refreshes screen → queries database → sees new order. Gap: 30 seconds to 5 minutes depending on refresh interval.

MedBrain uses NATS JetStream for event-driven workflows:

  Doctor orders CBC
  → LAB_ORDER_CREATED event emitted
  → Lab module subscribes → new order appears instantly on lab screen
  → Billing subscribes → charge auto-created (₹200 for CBC)
  → Queue subscribes → phlebotomist notified on mobile
  → SMS subscriber → patient SMS "sample collection at Counter 3"
  → Meilisearch subscriber → search index updated
  → Audit subscriber → access logged in PostgreSQL

Events are persistent (JetStream). If any subscriber is down, events are stored and replayed when the subscriber recovers. Zero data loss.

Event categories:
  PATIENT.*       — created, updated, merged, admitted, discharged
  ENCOUNTER.*     — started, updated, closed
  LAB.*           — ordered, collected, processing, resulted, verified, critical
  PRESCRIPTION.*  — created, dispensed, returned
  BILLING.*       — charged, invoiced, paid, refunded, claimed
  VITALS.*        — recorded, abnormal_detected
  OT.*            — scheduled, started, completed, cancelled
  NOTIFICATION.*  — sent, delivered, read, escalated
  ABAC.*          — access_granted, access_denied, break_glass, conflict_detected

2.7 Phase Implementation
────────────────────────

Phase 1 (April 2026 — Base demo):
  PostgreSQL only for all data. YottaDB architecture designed but not deployed.
  Simple polling/WebSocket for real-time instead of NATS.
  PostgreSQL full-text search instead of Meilisearch.
  Proves the product works, gets user feedback.

Phase 2 (May-June 2026):
  Add YottaDB for clinical data (patient records, encounters).
  Add NATS JetStream for event-driven workflows.
  Add Meilisearch for fuzzy search.
  Migrate clinical data from PostgreSQL to YottaDB.
  PostgreSQL retains billing, audit, analytics, ABAC.

Phase 3 (July+ 2026):
  Add Octo SQL layer on YottaDB for analytics.
  Add YottaDB replication for DR/HA.
  Full dual-database architecture as designed.


3. AUTHORIZATION MODEL — ABAC WITH DATA ACCESS LIFETIMES
═══════════════════════════════════════════════════

3.1 Why Not Simple RBAC
────────────────────────

Traditional HMS uses Role-Based Access Control: Doctor = can see patient records. Nurse = can see vitals. Admin = can see everything.

This is insufficient for hospital security because:
- A dermatologist should NOT see psychiatry notes
- A doctor from Ward A should NOT browse Ward B patients casually
- A nurse on day shift should NOT access records at 3 AM
- A lab tech who processes HIV test should NOT see patient address
- Access should EXPIRE after clinical context ends (discharge, shift end)
- Emergency break-glass needs different rules than routine access

MedBrain implements Attribute-Based Access Control (ABAC) — a single authorization model that answers:

  WHO (role + department + specific user)
  can do WHAT (read / write / create / delete / print / export)
  on WHICH resource (module + patient + field pattern)
  UNDER WHAT CONDITIONS (assigned doctor? emergency? referral? same ward?)
  FOR HOW LONG (2 hours? 15 days? until discharge?)
  VALIDATED BY (OTP? admission event? break-glass? supervisor approval?)
  REVOKED WHEN (discharge? shift end? referral period expires?)
  WITH WHAT AUDIT (standard? elevated? mandatory 24hr review?)

3.2 ABAC Policy Structure
─────────────────────────

All policies stored in a single PostgreSQL table:

  abac_policies
  ├── id (PK)
  ├── role_pattern          ("Dentist", "IPD_Doctor", "Lab_Tech_*")
  ├── department_pattern    ("DENTAL", "MEDICINE", "*")
  ├── resource_pattern      ("patient.*", "patient.clinical.*", "lab.results")
  ├── field_pattern         ("*", "-psychiatry_notes,-hiv_status", "vitals_only")
  ├── action                (READ, WRITE, CREATE, DELETE, PRINT, EXPORT)
  ├── condition_type        (ASSIGNED, ADMITTED, EMERGENCY, REFERRAL, CONSENT)
  ├── data_access_lifetime  (interval: '2 hours', '15 days', 'shift_duration')
  ├── validation_method     (OTP, AUTO_ON_ADMISSION, BREAK_GLASS, SUPERVISOR, PATIENT_CONSENT)
  ├── revocation_trigger    (DISCHARGE, SHIFT_END, REFERRAL_EXPIRY, MANUAL)
  ├── audit_level           (STANDARD, ELEVATED, MANDATORY_REVIEW)
  ├── is_active             (boolean)
  └── created_at, updated_at

3.3 4-Layer ABAC Evaluation
───────────────────────────

Every API request passes through four authorization layers:

Layer 1 — Module Access Check:
  Is the requested module active? (^MODULE status)
  Can this role access this module? (policy lookup)
  Result: ALLOW or DENY at module level

Layer 2 — Resource Access Check:
  Can this user access this specific patient?
  Is user assigned to this patient (OPD/IPD)?
  Is data access token still valid (not expired)?
  Was break-glass invoked (emergency)?
  Result: ALLOW or DENY at patient level

Layer 3 — Field-Level Filtering:
  Which fields can this user see?
  Apply field_pattern: hide psychiatry notes from dentist,
  hide HIV status from non-treating staff,
  hide Aadhaar from non-registration staff
  Result: Filtered response (fields removed/masked)

Layer 4 — Audit Logging:
  Every access logged to PostgreSQL audit_log table:
  who, what, when, from_where (IP), patient_id, action, result
  ELEVATED audit: additional context captured
  MANDATORY_REVIEW: flagged for supervisor review within 24 hours

Evaluation time: <5ms (Redis-cached policies, hot path optimized).

3.4 Data Access Lifetime — Context-Aware Sessions
──────────────────────────────────────────────────

This is NOT about login sessions. Users stay logged in normally. Data Access Lifetime controls how long a user can ACCESS PATIENT DATA after validation.

Clinical Context       | Data Access Lifetime | Validation Method      | Revocation Trigger
-----------------------|---------------------|------------------------|--------------------
OPD Consultation       | 1-2 hours           | OTP once at start      | Time expiry
IPD Assigned Doctor    | 15 days             | Auto on admission      | Discharge event
IPD Covering Doctor    | 8 hours (= shift)   | OTP once per shift     | Shift end
Emergency/Trauma       | 12 hours            | Break-glass, NO OTP    | Time expiry
Lab Result View        | 30 minutes          | Auto (ordered=access)  | Time expiry
Consent-gated (HIV)    | 24 hours            | Patient OTP/signature  | Time expiry
Referral Cross-dept    | 48 hours            | Referring doc grants   | Referral expiry
MBBS Student Posting   | Posting duration     | Auto on posting assign | Posting end
PG Resident Rotation   | 3-6 months          | Auto on rotation start | Rotation end

How it works:
1. Doctor logs in → normal session (JWT, 15-min access token + 7-day refresh)
2. Doctor opens OPD patient → system checks: does doctor have active data access token for this patient?
3. No token → prompt OTP validation (one-time for this consultation)
4. OTP validated → data access token created in Redis with 2-hour TTL
5. Doctor sees patient record, writes notes, orders labs — all within 2-hour window
6. After 2 hours → data access token expires → doctor still logged in but needs re-validation to access this patient's data again
7. For IPD: admission event auto-creates 15-day token → revoked on discharge event

Emergency override: Break-glass bypasses OTP, grants 12-hour access with maximum audit level and mandatory 24-hour review by Medical Superintendent.

Storage: Data access tokens in Redis with TTL. Audit trail in PostgreSQL. All configurable in master files — hospital admin can change durations (2hr → 3hr) but CANNOT change validation methods (OTP requirement, break-glass audit level). Validation methods are security invariants hardcoded in Rust — the compiler prevents weakening.

3.5 Policy Examples
───────────────────

Example 1 — OPD Dentist:
  Role: Dentist
  Department: DENTAL
  Resource: patient.clinical.dental.*
  Action: READ, WRITE
  Lifetime: 2 hours
  Validation: OTP once
  Field filter: dental records + basic demographics only
  Cannot see: psychiatry notes, HIV status, financial data

Example 2 — IPD Assigned Doctor:
  Role: IPD_Doctor
  Department: MEDICINE
  Resource: patient.* (full record)
  Action: READ, WRITE, ORDER
  Lifetime: 15 days
  Validation: Auto on admission (no OTP needed)
  Revocation: Discharge event
  Field filter: all clinical fields except psychiatry (unless psychiatrist)

Example 3 — Emergency Break-Glass:
  Role: Any doctor
  Department: Any
  Resource: patient.* (full record)
  Action: READ, WRITE, ORDER
  Lifetime: 12 hours
  Validation: Break-glass button (NO OTP, NO delay)
  Audit: MANDATORY_REVIEW within 24 hours
  Revocation: 12 hours or manual

Example 4 — MBBS Student:
  Role: MBBS_Student
  Department: Assigned posting department
  Resource: patient.clinical.anonymized
  Action: READ only
  Lifetime: Posting duration (15 days to 1 month)
  Validation: Auto on posting assignment
  Field filter: Anonymized — sees "45/M, Dengue Fever, Ward 2B, Bed 12" but NOT name, phone, address, Aadhaar, billing
  Revocation: Posting end — zero access after


4. ENCRYPTION ARCHITECTURE — 7 LAYERS
═══════════════════════════════════════════════════

4.1 Compliance Context
──────────────────────

DISHA (Digital Information Security in Healthcare Act, 2018 draft) was never passed as a separate law — its principles were subsumed into DPDPA 2023 (Digital Personal Data Protection Act). Actual mandates from DPDPA + NABH + ABDM combined:

- HTTPS for all data transmission
- Password hashing (no plaintext storage)
- Role-based access control with audit trail
- Encryption at rest for stored health data
- Consent management for data processing
- Breach notification within 72 hours
- Right to erasure (data deletion on request)
- Data processing purpose limitation

MedBrain exceeds all requirements with 7 encryption layers.

4.2 Layer 1 — Transit Encryption (Browser to Server)
─────────────────────────────────────────────────────

- TLS 1.3 mandatory (TLS 1.2 minimum for legacy clients)
- HSTS headers (Strict-Transport-Security)
- Certificate pinning on mobile app (React Native)
- mTLS for ABDM/TPA API integrations (mutual certificate verification)
- HTTP/2 for multiplexed connections

4.3 Layer 2 — Transit Encryption (Server to Databases)
──────────────────────────────────────────────────────

- YottaDB: IN-PROCESS — no network encryption needed (security advantage; there is no wire to tap)
- PostgreSQL: TLS with sslmode=verify-full (certificate verified)
- Redis/Dragonfly: TLS enabled for cache connections
- Meilisearch: HTTPS with API key authentication
- NATS: mTLS between all NATS nodes and clients
- MinIO: S3-compatible TLS for object storage

4.4 Layer 3 — At Rest Encryption (Database Files)
──────────────────────────────────────────────────

- YottaDB: Built-in Data-At-Rest (DAR) encryption — AES-256 with plugin architecture, per-database keys, FIPS 140-2 compliant mode available. Encryption is transparent to application code.
- PostgreSQL: Transparent Data Encryption (TDE) where available, or LUKS full-disk encryption
- LUKS: Full disk encryption on server storage volumes
- MinIO: Server-Side Encryption (SSE-S3) for all stored objects
- Backups: Encrypted with separate backup encryption key

4.5 Layer 4 — Field-Level Encryption (Application Layer)
────────────────────────────────────────────────────────

Sensitive fields encrypted BEFORE writing to any database. Even a DBA with full database access cannot read encrypted fields.

Encrypted fields (prefix "enc:v1:..."):
  PII: Aadhaar number, phone number, email, full address (street level)
  Clinical sensitive: HIV status, psychiatry notes, substance abuse records, STI results, abortion records
  Financial: Bank account, insurance policy number, payment card details

Different encryption keys for different sensitivity levels:
  KEK_PII → encrypts Aadhaar, phone, email
  KEK_CLINICAL → encrypts HIV status, psychiatry notes
  KEK_FINANCIAL → encrypts bank accounts, card details

Configuration in master files:
  ^MF("ENCRYPTION","RULES","patient.aadhaar") = "PII"
  ^MF("ENCRYPTION","RULES","patient.hiv_status") = "CLINICAL"
  ^MF("ENCRYPTION","RULES","billing.bank_account") = "FINANCIAL"

4.6 Layer 5 — Application Security
───────────────────────────────────

- Password hashing: Argon2id (64MB memory cost, 3 iterations, 4 parallelism) — resistant to GPU/ASIC attacks
- JWT: Ed25519 signed (not HMAC), 15-minute access tokens, 7-day refresh tokens
- MFA: TOTP (Google Authenticator compatible) mandatory for admin roles
- Session binding: JWT bound to IP + device fingerprint, invalidated on IP change
- Rate limiting: per-user, per-endpoint, configurable (e.g., 100 req/min for API, 5 attempts for login)
- CORS: strict origin whitelist
- CSRF: double-submit cookie pattern
- Input validation: Rust's type system + validator crate — invalid data cannot reach business logic
- SQL injection: impossible with sqlx compile-time query checking
- XSS: React's default escaping + Content-Security-Policy headers

4.7 Layer 6 — Per-Patient & Per-User Encryption
────────────────────────────────────────────────

Key Hierarchy:

  Level 1: Master Key (MK)
    → Stored in HashiCorp Vault or HSM
    → Shamir secret sharing: split into 3 parts, any 2 reconstruct
    → Rotated annually

  Level 2: Key Encryption Keys (KEK)
    → Per category: KEK_PII, KEK_CLINICAL, KEK_FINANCIAL
    → Encrypted by Master Key
    → Rotated quarterly (background re-encryption of PDKs)

  Level 3: Patient Data Keys (PDK)
    → Per patient: generated at registration
    → AES-256, unique per patient
    → Encrypted by appropriate KEK
    → If one patient's key compromised → only that patient affected

  Level 4: Field-Level Derived Keys
    → HKDF derived from PDK per field type
    → Never stored — derived on demand
    → Different key for Aadhaar vs phone vs HIV status

Per-User Encryption:
  User Encryption Key (UEK) per staff member
  → Protects personal staff data
  → Digital signature keypair (Ed25519) for prescriptions/documents
  → Rotated on password change

Consent-Based Encryption (HIV/Psychiatry):
  Patient Consent Key split into:
  → Part A: system-held (stored encrypted in vault)
  → Part B: derived from patient consent token (OTP/biometric)
  → Both parts needed to decrypt → patient must actively consent
  → Revocable: patient can revoke consent, Part B invalidated

Performance impact: Total encryption overhead ~0.7ms per patient record access (PDK fetch 0.5ms cached in Redis, decrypt 0.001ms AES-NI hardware, derive keys 0.003ms HKDF, decrypt fields 0.005ms, ABAC check 0.2ms) on top of ~2.5ms baseline = ~3.2ms total. At scale (420-bed hospital): <2% CPU, <50MB memory for cached keys.

Key rotation:
  Master Key: annual (re-encrypt KEKs only — minutes)
  KEK: quarterly (background re-encrypt PDKs — hours, zero downtime)
  PDK: on-demand per patient if compromised
  UEK: on password change
  All rotation is zero-downtime — old and new keys coexist during transition.

4.8 Layer 7 — Audit & Monitoring
─────────────────────────────────

- Every data access logged: who, what, when, from where, result
- Every encryption/decryption operation logged
- Failed access attempts trigger alerts
- Break-glass events flagged for mandatory review
- Tamper-evident audit log (hash chain, each entry references previous hash)
- Audit log in PostgreSQL, replicated to separate audit database
- Retention: minimum 7 years (medicolegal requirement), configurable per hospital

4.9 Implementation Phases
─────────────────────────

Phase 1 (April 2026 — Base demo):
  HTTPS, Argon2id password hashing, RBAC/ABAC, audit log, LUKS disk encryption, PostgreSQL ssl=on, session timeout, JWT with Ed25519.
  Effort: 1-2 days configuration.
  Result: Passes NABH/DPDPA audit. Better than 95% of Indian HMS vendors.

Phase 2 (3-6 months post-launch):
  MFA for admin roles, field-level encryption for Aadhaar, digital signatures for prescriptions, ABDM integration with mTLS, break-glass logging with mandatory review.
  Effort: 2-3 weeks development.

Phase 3 (Scale — 6-12 months):
  Per-patient keys, consent-based encryption for HIV/psychiatry, HSM integration, key rotation automation, Shamir secret sharing for Master Key, vault deployment.
  Effort: 1-2 months development.
  Result: Epic-level security. No Indian HMS competitor has this.


5. NOTIFICATION PRIVACY — NUMBERS ONLY
═══════════════════════════════════════════════════

5.1 Three Rules (Security Invariants)
─────────────────────────────────────

Rule 1 — Security invariants are HARDCODED:
  OTP requirements, encryption methods, consent flows, break-glass audit levels — these are compiled into the Rust binary. Hospital admin CANNOT weaken them. Admin can adjust durations (2hr → 3hr) but cannot remove OTP requirement. The compiler enforces this.

Rule 2 — Public channels show NUMBERS ONLY:
  TV displays, intercom, SMS, public queue boards — no patient names, no diagnosis, no clinical information. Token numbers, bed numbers, counter numbers only.

Rule 3 — Private screens show full data, ABAC filtered:
  Logged-in EMR screens show full names + clinical data, filtered by ABAC policies based on user's role, department, and data access token.

5.2 Notification Content Levels
───────────────────────────────

PUBLIC (TV display, intercom, public queue board):
  Token/code numbers only.
  OPD queue: "Token D-042 → OPD Room 5"
  Lab queue: "Token L-018 → Counter 2"
  Intercom: "Doctor 200, Room 3, Token D-018" (employee code, not name)
  Emergency: "Code Blue, Zone 3, Bed 12" (standard codes, location only)
  NEVER: patient name, diagnosis, doctor name on public channels.

PRIVATE (SMS/WhatsApp to patient's registered number):
  UHID allowed, no clinical details.
  "Patient ID ACMS-2026-001234, your lab results are ready. Visit Counter 5."
  "Patient ID ACMS-2026-001234, your appointment is confirmed for 10:30 AM, OPD Room 3."
  NEVER: diagnosis, test names, drug names in SMS.

SECURE (authenticated EMR screens, in-app notifications):
  Full information, ABAC filtered.
  Doctor sees: "Ravi Kumar (ACMS-2026-001234) — CBC result: Hb 8.2 g/dL (LOW) — Critical alert"
  Nurse sees: same if authorized for this patient.
  Student sees: "Case #2026-MED-0342, 45/M — CBC result: Hb 8.2 g/dL (LOW)" (anonymized)

5.3 Intercom Code System
─────────────────────────

Hospitals use intercoms constantly. MedBrain provides coded communication:

  "Doctor 200, Room 3, Token D-018"
  → Doctor employee code 200, go to Room 3, regarding patient Token D-018

  "Code Blue, Zone 3, Bed 12"
  → Cardiac arrest, Zone 3 (ICU wing), Bed 12

  "Code Red, Zone 1, Floor 2"
  → Fire emergency, Zone 1, Floor 2

  "Housekeeping, Zone 2, Room 205"
  → Cleaning needed, Zone 2, Room 205

No names. No diagnosis. No clinical information on any public channel. Ever. This is a security invariant — not configurable by admin.


6. MASKED COMMUNICATION SYSTEM
═══════════════════════════════════════════════════

6.1 The Problem
───────────────

Nurse discovers critical lab result (Potassium 6.8 — life-threatening). Needs to call the ordering doctor immediately.

Current reality: Nurse looks up doctor's personal mobile number from a paper directory or WhatsApp group, calls personal phone. Problems — doctor's personal number exposed to all staff, no call logging, no context on screen when phone rings, no escalation if doctor doesn't answer, no record for audit.

6.2 MedBrain Masked Communication
──────────────────────────────────

Modeled after Uber/Ola driver-rider communication — neither party's personal number is exposed.

Flow:
1. Nurse taps "Call Ordering Doctor" on critical lab result screen
2. MedBrain knows who ordered the test (from lab order record)
3. System assigns a virtual number/SIP extension
4. Bridges the call — nurse sees "HOSPITAL CALL" not personal number
5. Doctor's phone shows call context on screen:
   "HOSPITAL — CRITICAL LAB
    From: Nursing Station 3B
    Re: Patient ACMS-2026-001234
    Alert: Potassium 6.8 mEq/L"
6. Call logged: patient context, duration, response time — for analytics

Neither nurse nor doctor sees the other's personal number. All calls are in hospital context with relevant clinical information displayed.

6.3 Routing Intelligence
────────────────────────

When nurse calls doctor through MedBrain:

  Doctor available (in OPD/ward)
  → Ring desk phone + mobile simultaneously

  Doctor in consultation + CRITICAL result
  → Interrupt with "URGENT" flag on screen
  → Option to acknowledge + call back within 5 minutes

  Doctor in surgery
  → Route to covering doctor (from duty roster)
  → If covering doctor unavailable → escalate to HOD

  Doctor off duty
  → Route to on-call doctor (from on-call schedule)
  → If no answer → escalate to HOD → Medical Superintendent

Auto-escalation timeline:
  0 min: Ring ordering doctor
  2 min: No answer → retry once
  4 min: No answer → route to covering doctor + SMS alert to ordering doctor
  8 min: No answer from covering → route to HOD + SMS to both doctors
  15 min: No answer from HOD → route to Medical Superintendent + flag incident

Every escalation step logged. Response times tracked for analytics. Patterns identified: "Dr. X averages 8-minute response to critical results" — useful for quality improvement.

──────────────────────────

------------|-------------------------------|-------------------

7. ADDRESS-BASED CONFLICT DETECTION
═══════════════════════════════════════════════════

7.1 The Problem
───────────────

Small town hospital reality: Staff and patients come from the same neighborhoods. A nurse from K. Pudur assigned to a patient from K. Pudur creates a privacy breach risk — even without malicious intent. Neighbors ask questions, gossip spreads, especially for sensitive conditions: HIV, pregnancy, psychiatric treatment, substance abuse, sexual health, domestic violence.

No other Indian HMS addresses this. MedBrain does.

7.2 Conflict Matching Levels
─────────────────────────────

Level 1 — Same street/ward:
  Action: BLOCK automatically
  Override: Medical Superintendent approval only, with documented reason
  Audit: Maximum — every access logged with elevated review

Level 2 — Same pincode/locality:
  Action: FLAG for supervisor
  Override: Supervisor must acknowledge and document reason
  Audit: Elevated

Level 3 — Same village (rural areas):
  Action: FLAG high priority
  Especially for: HIV, psychiatry, substance abuse, maternity, STI
  Override: Supervisor acknowledgment required

Level 4 — Same district:
  Action: LOG only, monitor pattern
  No automatic blocking, but tracked for audit

7.3 Trigger Points
──────────────────

Conflict detection runs automatically at:
- IPD admission: check ALL ward nurses against patient address
- OPD appointment: check assigned doctor and department staff
- Lab sample collection: especially for HIV, pregnancy, STI tests
- Counseling sessions: strictest — same pincode = BLOCK
- Home care visits: check visiting staff against patient area
- Shift change: re-evaluate new shift staff against all ward patients

7.4 Sensitive Case Escalation
─────────────────────────────

For sensitive conditions (HIV, psychiatry, substance abuse, sexual health, abortion, domestic violence), the conflict detection radius INCREASES:

Normal case:    Same street = block, same pincode = flag
Sensitive case: Same pincode = block, same taluk = flag, same village = absolute block

Even lab technicians who process samples (and never meet the patient) are checked. A lab tech from the same village processing an HIV test is a privacy risk — the tech knows the patient sent blood for HIV testing.

7.5 Edge Cases
──────────────

No alternate staff available:
  → Supervisor acknowledges "staffing constraint" in system
  → Elevated audit for all interactions
  → Flagged for post-review

Only specialist available + same street:
  → Cannot block (patient needs specialist care)
  → FLAG + elevated audit + chaperone from different area
  → Restrict access to minimum necessary fields (no address, no phone)
  → Time-limited access token

Staff member IS the patient's relative:
  → ABSOLUTE BLOCK, no override except Medical Superintendent
  → Even in emergency, assign different staff if any available
  → Mandatory incident report

7.6 Data Model
──────────────

Staff and patient addresses stored with:
  → Locality/pincode/district/taluk/village: plaintext (for matching)
  → Street address: encrypted (not needed for matching)

Conflict rules stored in ABAC authorization table with:
  → Case sensitivity level: NORMAL, SENSITIVE, RURAL_SENSITIVE
  → Action: BLOCK, FLAG, LOG
  → Override permission level: SUPERVISOR, HOD, MEDICAL_SUPERINTENDENT
  → Audit level: STANDARD, ELEVATED, MANDATORY_REVIEW

This is NOT a separate system — it is one more check in the ABAC evaluation flow (inserted between Layer 2 and Layer 3 of the 4-layer evaluation).

10.4 Reliability Architecture
─────────────────────────────

Why 99.9% is achievable with a small team:

Rust eliminates crash categories: No NullPointerException, no OutOfMemoryError (ownership system), no thread deadlocks (borrow checker), no GC pauses (no garbage collector). Rust services run for months without restart. This alone takes uptime from ~95% (typical Java HMS) to ~99.5%.

YottaDB eliminates database connection failures: In-process means no connection pool exhaustion, no "too many connections" error, no network timeout, no replication lag on primary. If the Rust process is alive, YottaDB is alive. They are the same process.

Graceful degradation: Support services can fail without taking down clinical care.
  PostgreSQL down → clinical care continues (YottaDB has patient data), billing paused
  Redis down → ABAC falls back to PostgreSQL lookup (slower but works)
  Meilisearch down → falls back to direct YottaDB lookup (exact search only)
  NATS down → writes succeed, notifications delayed, events buffered
  MinIO down → can't upload new files, existing data unaffected
  Only single point of failure: Rust+YottaDB process (same process)

Systemd watchdog: Process crashes (rare in Rust) → auto-restart in 1 second. Process hangs → watchdog kills + restarts in 30 seconds. YottaDB journal replay on restart → zero data loss, 2-5 second recovery.

NATS JetStream persistence: Events stored durably. If any subscriber is down, events replayed when it recovers. Lab result entered while billing is down → billing auto-creates charge when it comes back.

Offline mobile: React Native app with WatermelonDB caches last 48 hours of patient data. Server down → doctor's phone still works, reads cached data, writes offline, syncs when connection restores. Effective uptime from doctor's perspective: ~100%.

10.5 Deployment Options
───────────────────────

Single node (Phase 1, most hospitals): 1 server + systemd + watchdog. 99.9% achievable. Recovery: 1-30 seconds per incident. Cost: ₹5-8K/month.

Active-passive (Phase 2, large hospitals): Primary + standby with YottaDB replication + PostgreSQL streaming replication. Health check failure → DNS failover (30-60 seconds). 99.95% achievable. Cost: ₹15-20K/month.

Active-active (Phase 3, hospital chains): Both nodes serve traffic, load balanced. One node dies → zero visible downtime. 99.99% achievable. Cost: ₹30-50K/month.
